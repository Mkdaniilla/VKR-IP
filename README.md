# VKR-IP

`VKR-IP` — веб-система для учета, анализа и оценки объектов интеллектуальной собственности. Проект объединяет реестр объектов ИС, модуль стоимостной оценки, генерацию документов, базу знаний и контроль дедлайнов в одном интерфейсе.

## Что умеет система

- вести реестр объектов интеллектуальной собственности;
- хранить связанные документы и материалы;
- рассчитывать стоимость объектов ИС с использованием `Relief from Royalty` и `DCF`;
- формировать PDF-отчеты по результатам оценки;
- генерировать типовые юридические документы;
- отображать аналитику и обзорные показатели на dashboard;
- поддерживать базу знаний по тематике ИС;
- работать с дедлайнами и связанными сущностями проекта.

## Архитектура

Проект состоит из трех основных частей:

- `frontend` — клиентская часть на `Next.js 15` и `React 19`;
- `backend` — серверная часть на `FastAPI`, `SQLAlchemy`, `Pydantic`;
- `PostgreSQL` — основная база данных.

Для локального и production-запуска используются `Docker` и `Docker Compose`. На сервере production-окружение работает через `Nginx`.

## Локальный запуск через Docker

Для локального запуска используется файл `docker-compose.yml`.

### 1. Подготовка переменных окружения

Скопируйте шаблоны и заполните значения:

```bash
cp .env.example .env
cp backend/.env.example backend/.env
```

Минимально необходимо задать:

- `DB_PASSWORD`
- `DATABASE_URL`
- `SECRET_KEY`
- `OPENROUTER_API_KEY`

```bash
docker-compose up --build -d
```

### 3. Доступ

- frontend: `http://localhost:3000`
- backend: `http://localhost:8000`
- backend docs: `http://localhost:8000/docs`

## Модуль оценки

Система оценки использует комбинированную логику:

- доходный подход через `Relief from Royalty`;
- дисконтирование денежных потоков (`DCF`);
- затратный подход как дополнительный ориентир;
- корректировки на юридические факторы, интервью и подтверждающие материалы;
- формирование диапазона значений `min / base / max`.

### Математическая модель оценки

В текущей реализации расчет строится поэтапно, а не одной формулой. Базовая логика соответствует комбинации доходного и затратного подходов с последующими корректировками.

#### 1. Определение годовой экономической выгоды

Годовая выгода от использования объекта ИС рассчитывается как:

```text
AnnualBenefit = Revenue × RoyaltyRate
```

где:

- `Revenue` — годовая выручка, связанная с активом;
- `RoyaltyRate` — ставка условного роялти, определяемая по отраслевым диапазонам и типу актива.

Если ставка роялти не задана вручную, система определяет ее автоматически:

- по отрасли (`industry`);
- по типу объекта ИС;
- по данным интервью с пользователем.

#### 2. Доходный подход через DCF

Доходная стоимость рассчитывается как чистая приведенная стоимость потока выгод:

```text
IncomeValue = Σ(AnnualBenefit / (1 + DiscountRate)^t),  t = 1..n
```

где:

- `AnnualBenefit` — годовая экономическая выгода;
- `DiscountRate` — ставка дисконтирования;
- `n` — горизонт прогнозирования в годах (`remaining_years`).

Ставка дисконтирования определяется по формуле:

```text
DiscountRate = BaseRate(industry) + RiskPremium
```

где:

- `BaseRate(industry)` — базовая ставка по отрасли;
- `RiskPremium` — надбавка за риск, определяемая через AI-анализ и интервью.

Для отдельных типов активов могут добавляться специальные риск-корректировки. Например, для `know-how` в текущей реализации добавляется дополнительная премия за риск раскрытия.

#### 3. Затратный подход

Затратная стоимость рассчитывается как:

```text
CostValue = RnD × 1.6
```

где:

- `RnD` — затраты на разработку или создание актива (`cost_rd`);
- коэффициент `1.6` в текущей модели учитывает инфляцию, налоги и предпринимательскую надбавку.

#### 4. Формирование базовой стоимости

После расчета доходной и затратной частей система формирует базовую стоимость:

```text
if Revenue > 1 000 000:
    Baseline = 0.9 × IncomeValue + 0.1 × CostValue
elif Revenue > 0:
    Baseline = 0.7 × IncomeValue + 0.3 × CostValue
elif CostValue > 0:
    Baseline = CostValue
else:
    Baseline = AI fallback estimate
```

Таким образом, в текущей production-логике веса не фиксированы как абстрактные `60/40`, а зависят от наличия и объема денежных потоков.

#### 5. Корректирующие коэффициенты

После расчета `Baseline` к стоимости применяются дополнительные множители:

```text
FinalValue =
    Baseline
    × LegalFactor
    × InterviewBonus
    × MarketReachFactor
    × JurisdictionFactor
    × BrandStrengthFactor
    × DirectionMultiplier
    × LicensingBonus
    × EncumbranceDiscount
    × ProtectionBonus
    × SubtypeMetricsFactor
    × ValuationPurposeMultiplier
```

Где:

- `LegalFactor` — коэффициент юридической устойчивости актива;
- `InterviewBonus` — бонус за подтвержденные ответы интервью;
- `MarketReachFactor` — поправка на географический охват;
- `JurisdictionFactor` — поправка на количество юрисдикций;
- `BrandStrengthFactor` — поправка на силу бренда;
- `DirectionMultiplier` — поправка на характер использования бренда;
- `LicensingBonus` — поправка на лицензионную активность;
- `EncumbranceDiscount` — дисконт за обременения;
- `ProtectionBonus` — бонус за подтвержденную защищенность;
- `SubtypeMetricsFactor` — поправка на дополнительные метрики подтипа;
- `ValuationPurposeMultiplier` — коэффициент цели оценки.

#### 6. Формирование диапазона оценки

Система возвращает не только одно итоговое значение, но и диапазон:

```text
ConfidenceSpread = max(0.05, 0.15 - 0.01 × EvidenceCount)
FinalMin = FinalValue × (1 - ConfidenceSpread)
FinalMax = FinalValue × (1 + ConfidenceSpread)
```

где:

- `EvidenceCount` — количество подтверждающих факторов и доказательств;
- чем больше подтвержденных данных, тем уже диапазон оценки.

#### 7. Evidence-based логика

В текущей реализации оценка действительно использует доказательный подход:

- ответы интервью сохраняются как подтвержденные факторы;
- связанные документы добавляются в `evidence_logs`;
- итоговый диапазон зависит от объема подтвержденной доказательной базы;
- отчет по оценке использует эти данные для интерпретации результата.

#### 8. Практический смысл модели

Текущая серверная версия проекта использует:

- `Relief from Royalty` как основу доходного метода;
- `DCF` для приведения будущих выгод к текущей стоимости;
- затратный подход как резервный и поддерживающий ориентир;
- систему поправок на правовую защиту, рыночный охват, силу бренда и качество доказательной базы.

Именно такая модель соответствует текущей реализации `backend/app/services/valuation_engine.py`, а не более упрощенной формуле из старых версий документации.

Основные файлы модуля:

- `backend/app/services/valuation_engine.py`
- `backend/app/services/valuation_report.py`
- `backend/app/routers/valuation.py`
- `frontend/pages/valuation/index.tsx`
