# Техническое задание на разработку системы `VKR-IP`

## 1. Общие сведения

### 1.1 Наименование проекта

`VKR-IP` — веб-платформа для учета, анализа и оценки объектов интеллектуальной собственности.

### 1.2 Назначение системы

Система предназначена для автоматизации работы с объектами интеллектуальной собственности, включая:

- ведение реестра объектов ИС;
- хранение связанных документов;
- расчет стоимости объектов ИС;
- генерацию юридических документов;
- контроль сроков и ключевых событий;
- предоставление аналитики через веб-интерфейс.

### 1.3 Цель разработки

Создать единую информационную систему, позволяющую пользователю работать с объектами интеллектуальной собственности в одном интерфейсе: от учета и анализа до оценки стоимости и подготовки документов.

## 2. Пользователи системы

Система ориентирована на следующие категории пользователей:

- специалисты по интеллектуальной собственности;
- юристы и патентные специалисты;
- аналитики и руководители проектов;
- администраторы системы;
- преподаватели и члены комиссии при демонстрации ВКР.

## 3. Функциональные требования

### 3.1 Учет объектов интеллектуальной собственности

Система должна обеспечивать:

- создание, редактирование и удаление карточек объектов ИС;
- хранение типа объекта, статуса, описания и служебных атрибутов;
- поддержку различных типов объектов, включая программное обеспечение, базы данных, товарные знаки, изобретения и иные результаты интеллектуальной деятельности;
- отображение списка объектов и детальной информации по каждому объекту.

### 3.2 Оценка стоимости объектов ИС

Система должна обеспечивать:

- проведение интерактивного интервью с пользователем для сбора исходных данных;
- учет финансовых, правовых и рыночных факторов;
- расчет стоимости объекта с использованием подхода `Relief from Royalty` и `DCF`;
- формирование интерпретации результатов оценки;
- вывод результатов оценки в пользовательском интерфейсе.

### 3.3 Формирование отчетов

Система должна обеспечивать:

- формирование отчета по результатам оценки;
- включение в отчет исходных параметров, методологии и итоговых расчетов;
- фиксацию данных, использованных при оценке, для последующего анализа и проверки.

### 3.4 Генерация юридических документов

Система должна обеспечивать автоматическую генерацию типовых юридических документов, связанных с объектами ИС. В текущей версии поддерживаются следующие типы:

- досудебная претензия;
- соглашение о конфиденциальности (`NDA`);
- договор отчуждения исключительного права;
- лицензионный договор;
- исковое заявление;
- договор коммерческой концессии;
- договор залога исключительного права;
- уведомление о нарушении.

Для каждого документа система должна:

- использовать шаблон;
- подставлять данные по выбранному объекту ИС;
- формировать итоговый документ для пользователя;
- сохранять связь документа с соответствующим объектом ИС.

### 3.5 Панель аналитики

Система должна обеспечивать:

- отображение ключевых показателей по портфелю ИС;
- визуализацию структуры портфеля;
- отображение диаграмм по стоимости и составу объектов;
- предоставление обзорной информации для принятия решений.

### 3.6 Контроль сроков и событий

Система должна обеспечивать:

- учет значимых сроков по объектам ИС;
- отображение ближайших событий;
- использование модуля контроля дедлайнов в общей логике системы.

### 3.7 База знаний

Система должна обеспечивать:

- хранение тематических материалов по интеллектуальной собственности;
- просмотр и использование базы знаний через интерфейс системы;
- поддержку сценариев поиска и справочного использования данных.

## 4. Нефункциональные требования

### 4.1 Требования к архитектуре

Система должна быть реализована как клиент-серверное веб-приложение и включать:

- frontend на `Next.js`;
- backend на `FastAPI`;
- базу данных `PostgreSQL`;
- контейнеризацию через `Docker` и `Docker Compose`.

### 4.2 Требования к надежности

Система должна:

- корректно запускаться в контейнерной среде;
- поддерживать проверку работоспособности backend через health endpoint;
- обеспечивать возможность перезапуска сервисов без ручной перенастройки кода.

### 4.3 Требования к безопасности

Система должна:

- использовать переменные окружения для секретов и конфиденциальных параметров;
- не хранить рабочие ключи и пароли в исходном коде;
- ограничивать доступ к инфраструктурным данным только авторизованным администраторам.

### 4.4 Требования к производительности

Система должна:

- корректно работать в современном браузере на настольных устройствах;
- обеспечивать приемлемое время отклика для типовых пользовательских операций;
- позволять работать с основными страницами без критических ошибок рендеринга.

### 4.5 Требования к сопровождаемости

Система должна:

- иметь разделение frontend- и backend-части по зонам ответственности;
- хранить конфигурацию развертывания в репозитории;
- содержать документацию по запуску и структуре проекта.

- `Docker`
- `Docker Compose`
- `Nginx`

## 6. Требования к развертыванию

### 6.1 Локальное окружение

Для локального запуска должны быть доступны:

- frontend на порту `3000`;
- backend на порту `8000`;
- база данных `PostgreSQL`.

### 6.2 Production-окружение

Production-версия должна:

- работать через `Nginx`;
- использовать отдельный production-конфиг;
- поддерживать запуск через `docker-compose.prod.yml`;
- использовать переменные окружения для URL, ключей и параметров подключения.

## 7. Требования к репозиторию

Репозиторий проекта должен:

- содержать актуальную основную ветку `main`;
- включать код, соответствующий текущему состоянию проекта;
- содержать текстовую проектную документацию;
- не содержать рабочих секретов, токенов, паролей и серверных артефактов.

## 8. Критерии приемки

Система считается соответствующей ТЗ, если:

- проект успешно запускается по инструкции из репозитория;
- frontend корректно собирается и отображает основные страницы;
- backend запускается и отвечает на базовые запросы;
- модуль оценки работает и выводит результат;
- модуль генерации документов доступен в интерфейсе;
- в репозитории присутствует актуальное текстовое техническое задание на русском языке.

## 9. Перспективы развития

В дальнейшем система может быть расширена за счет:

- ролевой модели доступа;
- углубленной аналитики по портфелю ИС;
- расширения набора шаблонов документов;
- повышения покрытия тестами;
- дополнительной интеграции с внешними правовыми и аналитическими сервисами.
